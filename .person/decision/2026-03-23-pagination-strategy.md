# 게시글 페이징 번호 및 유효성 검사 전략

- 날짜: 2026-03-23
- 참여: 사용자 + Gemini CLI

## 배경
게시글 페이징(Step 3) 구현 중, 개발자 중심의 0-기반 인덱스와 사용자 중심의 1-기반 페이지 번호 사이의 간극을 해결하고, 잘못된 페이지 요청에 대한 방어 로직을 설계함.

## 핵심 논의
1. **페이지 시작 번호 결정**: SQL `OFFSET` 공식(page * size)에는 0-기반이 유리하지만, 사용자 경험(UX)을 위해 주소창에는 1-기반 번호를 노출하기로 함.
2. **로직의 위치**: "문지기" 역할을 하는 컨트롤러에서 외부의 1-기반 숫자를 내부의 0-기반 숫자로 변환(-1 보정)하고 유효성을 검사하는 것이 서비스의 순수성 유지에 유리함을 확인.
3. **유효성 검사 방식**: 
    - **최솟값 미달(page < 1)**: 주소창과 데이터의 정합성을 위해 `?page=1`로 **리다이렉트** 처리.
    - **최댓값 초과**: 데이터가 없는 페이지 요청 시 서비스에서 **404 Exception**을 발생시켜 명확한 에러 상태를 전달.

## 결론
1. **사용자 친화적 1페이지 시작**: 주소창은 `?page=1`부터 시작하며, 컨트롤러가 이를 받아 `page - 1`로 서비스에 넘긴다.
2. **컨트롤러에서 리다이렉트**: 사용자가 `0` 이하의 페이지를 요청하면 강제로 `1`페이지 주소로 이동시킨다.
3. **서비스에서 404 예외**: 조회된 결과 리스트가 비어있을 경우(size == 0), 요청한 페이지가 존재하지 않는 것으로 간주하고 예외를 던진다. (추후 Step 4에서 `COUNT(*)` 도입 시 전체 페이지 수 기반으로 고도화 예정)
